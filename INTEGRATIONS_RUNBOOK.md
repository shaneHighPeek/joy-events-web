## 1) Current Architecture (No UI details)

### Data ingestion paths
1. **Static vacuum file**
   - Generated by Python vacuum pipeline
   - Output: `joy-events/data/events.json`
   - Copied into web app: `joy-events-app/src/data/events.json`

2. **Runtime API augmenters** (`/api/events`)
   - Ticketmaster (runtime, via env key)
   - Eventbrite (runtime, via env token)
   - Eventfinda (runtime, via API username/password; wiring complete, needs credentials)
   - Approved community submissions from Supabase

3. **Final feed**
   - `/api/events` merges static + runtime sources
   - Applies location and vibe filtering
   - Returns normalized event cards

## 4) Community Submission Workflow

1. User submits event in-app
2. Event inserted into `user_submitted_events` with `status = pending`
3. Approval email sent to admin
4. Admin clicks one-click approve link
5. Status updated to `approved`
6. Approved event appears in live `/api/events` feed

Supabase table used:
- `user_submitted_events`

## 6) Known Risks / Notes

3. **Ticketek** requires partnership/API or advanced browser strategy.
4. Runtime source counts can fluctuate by external provider availability.
